# 2.3 배포 & 시나리오 점검

> 고도화된 앱을 최종 배포하고, 실사용 시나리오로 검증합니다

* * *

## 🎯 한 줄 정의

**최종 배포 = 고도화된 앱을 인터넷에 올려서, 실제 업무 시나리오로 테스트하는 것**

* * *

## 왜 필요한가?

개발 환경에서 잘 되는 것과 **배포된 상태에서 잘 되는 것**은 다를 수 있습니다. 또한 본인이 아닌 **다른 사람이 처음 접속**했을 때도 문제없이 동작하는지 확인해야 합니다.

| 배포 전 테스트만 | 배포 후 시나리오 점검 |
| --- | --- |
| "내 화면에서는 잘 돼" | 링크로 접속해도 잘 됨 확인 |
| 핵심 기능만 테스트 | 처음~끝 전체 흐름 테스트 |
| 내가 쓸 줄 아니까 OK | **다른 사람도** 헤매지 않는지 확인 |

* * *

## 📋 따라하기 (20분)

### Step 1: 최종 배포 (3분)

Replit에서 **Deploy** 버튼을 클릭하면 이 화면이 나옵니다. 정말 이게 끝입니다:

![Replit Publish 약식 화면](/images/replit-publish-simple.jpg)

1. **도메인 이름** 확인 (자동 생성, Available이면 OK)
2. **접근 권한** 선택 (Public 추천)
3. 파란색 **Publish** 버튼 클릭 → 끝!

접근 권한은 구글 드라이브 공유 설정이랑 똑같습니다:

![접근 권한 드롭다운](/images/replit-publish-access-dropdown.jpg)

고급 설정이 필요하면 **Adjust settings**를 펼치면 됩니다:

![Replit Publish 화면 상단](/images/replit-publish-top.jpg)

| 항목 | 설명 |
|------|------|
| **Publish to domain** | 내 앱의 URL 주소. `프로젝트명.replit.app`으로 자동 생성 |
| **Who can access** | 접근 권한 설정 (아래 이미지 참고) |
| **Feedback widget** | 켜면 앱 구석에 피드백 버튼 생김 — 데모 때 유용! |
| **Deployment type** | **Autoscale**(추천) — 방문자 수에 따라 자동 확장/축소 |
| **Machine configuration** | 서버 사양. 해커톤에서는 기본값(2 vCPU/4GB)이면 충분 |

![Replit Publish 화면 하단](/images/replit-publish-bottom.jpg)

| 항목 | 설명 |
|------|------|
| **Deployment secrets** | API 키 같은 비밀 정보. 개발 환경 Secrets와 자동 동기화 |
| **Security scan** | 배포 전 보안 취약점 자동 검사 (Beta) |
| **Production database** | 체크하면 프로덕션 전용 DB 자동 생성 |
| **Costs included in Core** | Core 구독에 배포 비용 포함! |

**접근 권한 설정** — 구글 드라이브 공유 설정이랑 똑같습니다:

![Replit 접근 권한 설정](/images/replit-publish-access.jpg)

| 옵션 | 설명 |
|------|------|
| **Public** | URL 아는 사람 누구나 접속 (해커톤 데모용 추천!) |
| **Restricted with password** | 비밀번호 입력해야 접속 (Enterprise 전용) |
| **Workspace only** | 같은 워크스페이스 멤버만 접속 |
| **Only you** | 나만 접속 가능 |

> **핵심**: 서버, 도메인, SSL, 스케일링... 아무것도 몰라도 됩니다. **Publish 버튼 하나면 전 세계 누구나 접속할 수 있는 웹서비스가 됩니다.**

1. 설정 확인 (대부분 기본값 그대로 OK)
2. 파란색 **Publish** 버튼 클릭
3. 1~2분 기다리면 배포 완료!
4. **배포 URL 복사**

> 💡 이미 Part 1에서 배포했다면, Redeploy로 최신 버전으로 업데이트됩니다.

### Step 2: 실사용 시나리오 점검 (7분)

배포된 URL을 **시크릿 모드(Ctrl+Shift+N)**에서 열어서 테스트합니다. 시크릿 모드를 쓰는 이유는 **로그인, 캐시 없이 순수하게** 테스트하기 위함입니다.

#### 시나리오 체크리스트

```
기본 동작:
□ URL 접속 시 앱이 정상 로딩되는가?
□ 로딩 시간이 10초 이내인가?
□ 첫 화면에서 "뭘 해야 하는지" 알 수 있는가?

핵심 흐름:
□ 입력 → 처리 → 출력 전체 흐름이 에러 없이 동작하는가?
□ 결과물이 예상대로 나오는가?
□ 결과를 복사/저장할 수 있는가?

엣지 케이스:
□ 빈 입력 시 안내 메시지가 나오는가?
□ 매우 긴 입력도 처리 가능한가?
□ 뒤로가기/새로고침해도 앱이 정상인가?

모바일:
□ 핸드폰으로 접속 시 화면이 깨지지 않는가?
□ 터치로 버튼 클릭이 가능한가?
```

### Step 3: 동료 테스트 (5분)

옆 참가자 2~3명에게 **설명 없이** 앱 링크만 보내고 써보게 합니다.

```
관찰 포인트:
1. 어디를 먼저 클릭하는가? → UX 직관성 확인
2. 어디서 멈추는가? → 안내 문구 필요한 지점 발견
3. "이거 어떻게 해요?" 질문이 나오는가? → 개선 포인트
```

### Step 4: 마지막 수정 (5분)

시나리오 점검에서 발견된 **치명적 문제 1개만** 수정합니다.

```
⚠️ 이 시점에서의 수정 원칙:
- 동작 안 하는 것만 수정 (디자인 변경 X)
- 1개만 수정 (여러 개 동시 X)
- 수정 후 반드시 재배포
```

## 💡 데모 준비

### 30초~1분 데모 구조

```
[5초]  "저는 ___를 만들었습니다"
[10초] "___라는 문제를 해결합니다"
[30초] (화면 시연) 핵심 흐름 보여주기
        — 입력 넣기 → 버튼 클릭 → 결과 보여주기
[15초] "다음 2주간 ___를 개선할 예정입니다"
```

> 💡 **데모 팁**: 실제 업무 데이터(또는 그에 가까운 예시)를 넣어서 시연하면 설득력이 훨씬 높아집니다. "lorem ipsum" 같은 더미 데이터는 피하세요.

### 데모 전 체크

```
□ 배포 URL이 정상 접속 가능
□ 시연용 입력 데이터 준비 완료
□ 핵심 흐름을 1분 내에 보여줄 수 있음
□ 개선 계획 3개 정리 완료
```
