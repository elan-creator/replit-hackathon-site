## 🎯 한 줄 정의

* * *

## 📋 오늘 한 것 정리

### 4시간 동안의 여정

| 시간 | 한 일 | 산출물 |
| --- | --- | --- |
| Part 0 | 시작 | 해커톤 목표 & Replit 소개 |
| Part 1 | 아이디어 → 문제정의 → 첫 MVP | 동작하는 앱 & 데모 링크 |
| 휴식 | 잠깐 쉬면서 머리 환기 | ☕ |
| Part 2 | 오류 수정 & 고도화 → 배포 | 안정적인 앱 & 최종 데모 링크 |
| Part 3 | 마무리 | 2주 실행 계획 & 데모 발표 |

### 오늘 얻은 것

- ✅ **실행 가능한 MVP 링크** — 바로 공유 가능하고, 업무에 적용 가능
- ✅ **바이브코딩 경험** — AI에게 자연어로 설명해서 앱을 만드는 전체 프로세스 체득
- ✅ **프롬프트 작성 노하우** — 5가지 요소 + 단계별 요청 + 복붙 가능한 템플릿
- ✅ **4가지 핵심 역량** — 프롬프트 설계, 오류 수정 루프, 버전/롤백 관리, 배포 후 검증
- ✅ **"나도 개발할 수 있다"는 자신감** — 아이디어를 직접 현실로 만드는 힘

> 💡 여러분은 오늘, 개발팀의 도움 없이도 아이디어를 실현할 수 있다는 것을 증명했습니다. 이 경험은 업무 방식 자체를 바꿀 수 있는 잠재력을 가집니다.

* * *

## 🚀 2주 실행 계획

오늘 만든 앱을 그대로 방치하면 아깝습니다. 2주 동안 꾸준히 개선하고 팀에 확산시켜보세요.

### Week 1: MVP 안정화 & 팀 내 시범 적용

- **Day 1-2**: 오늘 받은 피드백 중 미반영 항목 처리 (특히 핵심 기능 관련)
- **Day 3-4**: 실제 업무 데이터(또는 최대한 유사한 데이터)를 넣어서 테스트
- **Day 5**:  팀원 1~2명에게 시범 사용 요청 및 초기 피드백 수집

### Week 2: 확산 준비 & 적용 시작

- **Day 6-7**: 시범 사용 피드백 반영 및 UX 개선 (쉬운 것 위주)
- **Day 8-9**: 팀 내 사용 가이드 작성 (간단하게, 템플릿 제공 등)
- **Day 10**: 팀 내 공유 및 적용 시작 (정식 사용)

### 실행 계획 작성

아래 양식을 채워서 여러분의 2주 계획을 구체화해보세요:

```
┌─────────────────────────────────────────┐
│ 🚀 2주 실행 계획                         │
├─────────────────────────────────────────┤
│                                          │
│ 앱 이름: ___________________________      │
│ 배포 URL: ___________________________     │
│                                          │
│ 💡 개선 항목 3개 (우선순위 순서대로):      │
│   1. ___________________________ (이번 주)  │
│   2. ___________________________ (이번 주)  │
│   3. ___________________________ (다음 주)  │
│                                          │
│ 👥 팀 내 확산 대상:                      │
│   - ___________________________ (누구에게?) │
│   - ___________________________ (언제부터?) │
│                                          │
│ 🙋 후속 코칭 필요 여부: Y / N            │
│                                          │
└─────────────────────────────────────────┘
```

> 💡 이 계획을 잘 세워두면, 해커톤으로 끝나는 것이 아니라 여러분의 실제 업무 효율을 높이는 데 기여할 수 있습니다. 2주 동안 꾸준히 관심을 가져주세요!

## 🎤 데모 타임

### 1인 1데모 (30초~1분, 핵심만 보여주세요)

발표 구조:

```
1. [5초]  "안녕하세요, 저는 [본인 이름]입니다. 
          오늘 [앱 이름]을 만들었습니다."
2. [10초] "이 앱은 [어떤 문제]를 해결합니다."
3. [30초] (화면 시연) 핵심 흐름 보여주기
          — 입력 넣기 → 버튼 클릭 → 결과 보여주기 (가장 극적인 부분)
4. [15초] "다음 2주간 [개선 항목 1], [개선 항목 2]를 개선할 예정입니다."
```

> 💡 **데모 팁**: 지루하게 설명하지 마세요. 화면을 보면서 "짠!" 하고 보여주는 것이 가장 효과적입니다. 실제 업무 데이터(또는 그에 가까운 예시)를 넣어서 시연하면 설득력이 훨씬 높아집니다.
