# 1.1 문제정의와 아이디어 스코프

> 만들 앱의 범위를 좁혀야 4시간 안에 완성할 수 있습니다

* * *

## 🎯 한 줄 정의

**문제정의 = "누구의 어떤 문제를 해결할 것인가"를 1장으로 정리하는 것**

* * *

## 왜 필요한가?

해커톤에서 가장 흔한 실패 원인은 **"너무 많이 만들려다가 아무것도 완성 못 하는 것"**입니다. 4시간은 생각보다 짧습니다. 기능 10개를 50%씩 만드는 것보다, 기능 3개를 100% 완성하는 게 훨씬 가치 있습니다.

| 스코프 없이 시작 😰 | 스코프 잡고 시작 ✨ |
| --- | --- |
| "일단 만들어보자" → 방향 잃음 | "이 기능 1개만 만들자" → 집중 |
| 50분 후 "뭘 만들고 있지?" | 50분 후 "동작하는 앱 완성!" |
| AI에게 모호한 프롬프트 → 엉뚱한 결과 | AI에게 명확한 지시 → 원하는 결과 |
| 토큰 낭비 + 시간 낭비 | 효율적인 빌드 |

**핵심**: 초반 스코프가 넓으면 **토큰과 시간 낭비가 커집니다.** Replit Agent에게 모호한 요청을 하면 AI도 헤매게 됩니다.

* * *

## 📋 따라하기: 문제정의 캔버스 (20분)

### Step 1: 대상 사용자 정의 (3분)

> "누가 이 앱을 쓰는가?"

구체적일수록 좋은 프롬프트가 됩니다.

```
❌ 모호한 사용자: "사람들", "회사원", "누구나"
✅ 구체적 사용자: "매주 팀 보고서를 정리하는 PM"

더 많은 예시:
- 고객 문의를 분류하는 CS 운영 담당자
- 회의록 액션아이템을 추적하는 팀장
- 매일 경쟁사 뉴스를 모니터링하는 마케터
- 프로젝트 일정을 취합하는 PMO
```

**작성 팁**: **"나 자신"도 훌륭한 대상 사용자입니다.** 내가 매일 귀찮아하는 일 = 최고의 아이디어. 본인이 직접 사용자이면 피드백 루프도 가장 빠릅니다.

### Step 2: 핵심 문제 정의 (3분)

> "어떤 문제를 해결하는가?"

문제를 한 문장으로 적어보세요:

```
"____________ 때문에 ____________ 하기가 어렵다"

예시:
- "매주 수작업으로 데이터를 정리" 때문에 "빠른 의사결정" 하기가 어렵다
- "고객 문의가 흩어져 있어서" "반복 질문에 빠르게 답변" 하기가 어렵다
- "회의록이 텍스트 덩어리라서" "누가 뭘 해야 하는지 파악" 하기가 어렵다
- "신규 입사자 질문이 매번 비슷해서" "같은 설명을 반복" 하기가 어렵다
```

> 💡 **문제를 못 찾겠으면?** 이번 주에 "아 또 이거 해야 돼?" 라고 느낀 순간을 떠올려보세요. 그게 문제입니다.

### Step 3: Must 기능 3개 정의 (10분)

> **이 단계가 가장 중요합니다.** 기능을 반드시 3개로 제한하세요.

**왜 3개인가?**
- 4시간 해커톤에서 현실적으로 완성 가능한 범위
- AI에게 명확한 프롬프트를 주려면 기능이 구체적이어야 함
- 3개면 핵심 사용 흐름(입력 → 처리 → 출력)을 구성할 수 있음

| 우선순위 | 기능 | 왜 필요한가 |
| --- | --- | --- |
| Must #1 | ____________ | 이게 없으면 앱이 아님 |
| Must #2 | ____________ | 핵심 사용 흐름에 필수 |
| Must #3 | ____________ | 사용성을 위해 최소한 필요 |

**실제 예시 — 회의록 액션아이템 추출 앱:**

| 우선순위 | 기능 | 왜 필요한가 |
| --- | --- | --- |
| Must #1 | 회의록 텍스트 입력창 | 입력이 없으면 시작 불가 |
| Must #2 | 액션아이템 자동 추출 | 앱의 핵심 가치 |
| Must #3 | 담당자별 그룹핑 + 복사 버튼 | 추출만 하면 쓰기 불편 |

**실제 예시 — 고객 FAQ 자동 답변:**

| 우선순위 | 기능 | 왜 필요한가 |
| --- | --- | --- |
| Must #1 | FAQ 데이터 입력/관리 | 답변의 기반 데이터 |
| Must #2 | 고객 질문 입력 | 매칭할 질문이 필요 |
| Must #3 | 적합한 답변 초안 생성 | 앱의 핵심 가치 |

### Step 4: 제외 범위 확인 (4분)

> **"이건 오늘 안 한다"를 명확히 적어두세요.** 빌드 중 "이것도 넣을까?" 유혹이 반드시 옵니다.

```
❌ 오늘 안 하는 것 (예시):
- 로그인/회원가입 기능
- 결제 기능
- 복잡한 대시보드
- 이메일/슬랙 자동 발송
- 모바일 완벽 최적화
- 데이터베이스 설계
- 여러 사용자 동시 접속
```

> 💡 **이 목록을 프롬프트에도 포함하세요.** "로그인 기능은 필요 없어"라고 명시하면 AI가 불필요한 기능을 만들지 않아서 시간을 아낄 수 있습니다.

## ⚠️ 흔한 실수 & 해결법

### 실수 1: 스코프가 너무 넓음

```
❌ "팀 전체 프로젝트 관리 도구를 만들겠다"
   → 칸반보드, 타임라인, 알림, 권한... 4시간으론 절대 불가능

✅ "회의록에서 액션아이템을 추출하는 도구를 만들겠다"
   → 입력, 추출, 표시 — 3개면 충분
```

**해결법**: "이 앱의 핵심 화면 1개만 그려봐"라고 스스로에게 물어보세요. 화면이 1개면 적정, 3개 이상이면 줄여야 합니다.

### 실수 2: 사용자가 불명확

```
❌ "모든 사람이 쓸 수 있는 앱"
✅ "우리 팀 PM이 매주 월요일에 쓰는 앱"
```

**해결법**: 사용자를 "한 사람"으로 특정하세요. 그 사람이 실제로 어떤 상황에서 쓰는지까지 상상하면 프롬프트 품질이 확 올라갑니다.

### 실수 3: 기능이 너무 많음

```
❌ Must 기능 7개 → AI 혼란 + 시간 부족
✅ Must 기능 3개 + "Nice to have" 4개
```

**해결법**: 4번째부터는 "Nice to have" 목록에 적어두세요. Part 2에서 시간이 남으면 추가합니다.
