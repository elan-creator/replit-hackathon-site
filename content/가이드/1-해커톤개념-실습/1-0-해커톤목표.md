# 1.0 해커톤 목표

> 아이디어를 4시간 내에 실행 가능한 앱으로 전환

* * *

## 🎯 한 줄 정의

**해커톤 = 짧은 시간 안에 아이디어를 동작하는 프로토타입으로 만드는 집중 빌드 세션**

* * *

## 왜 해커톤인가?

### 기존 업무 방식의 문제

```
아이디어 → 기획서 작성 → 디자인 시안 → 개발팀 의뢰 → 대기 → 수정 요청 → 대기 → ...
                                                        ↑
                                                여기서 3~6개월 소요
```

아이디어가 좋아도 **실행까지의 거리가 너무 멀어서** 대부분 기획서 단계에서 묻힙니다. 개발팀 리소스는 항상 부족하고, 우선순위에 밀리면 몇 달이고 대기합니다.

### 해커톤 방식

```
아이디어 → 프롬프트 → 동작하는 앱 → 피드백 → 개선
              ↑                          ↑
          30분 이내                  즉시 반영
```

**차이점**: 중간 단계를 과감히 생략하고, **동작하는 것**을 먼저 만듭니다. 완벽한 기획서보다 "돌아가는 앱 1개"가 더 많은 것을 알려줍니다.

### 실제 비교

| 항목 | 기존 방식 | 해커톤 방식 |
| --- | --- | --- |
| 아이디어 → 동작하는 앱 | 3~6개월 | **4시간** |
| 필요한 인원 | PM + 디자이너 + 개발자 | **나 혼자** |
| 필요한 도구 | 기획서, 디자인툴, IDE, 서버 | **브라우저 1개** |
| 실패 비용 | 수백~수천만원 | **4시간** |

* * *

## 🎯 오늘의 목표 3가지

### 1. 동작하는 MVP 링크 확보

- 아이디어가 아닌 **실제로 클릭해서 써볼 수 있는 앱**
- 누구에게든 링크 하나로 공유 가능
- "이거 써봐" 한 마디로 피드백 수집 시작

### 2. 바이브코딩 체험

- AI에게 자연어로 설명해서 앱을 만드는 경험
- "코딩을 몰라도 앱을 만들 수 있다"를 직접 확인
- 에러가 나도 당황하지 않고 AI에게 수정 요청하는 루프 학습

### 3. 2주 실행 계획 수립

- 해커톤 후 뭘 더 개선할지 구체적으로 정리
- "만들어 놓고 끝"이 아닌, **실무에 적용**하는 게 진짜 목표
- 팀 내 확산 대상까지 선정

* * *

## 📊 성공 기준

| 항목 | 기준 |
| --- | --- |
| **앱 동작** | 핵심 사용자 흐름 1개가 에러 없이 동작 |
| **배포 완료** | 링크를 타인에게 공유할 수 있는 상태 |
| **데모 가능** | 30초~1분 내 핵심 기능을 보여줄 수 있음 |
| **다음 단계 확정** | 개선할 항목 3개 + 2주 타임라인 |

> 💡 **"핵심 사용자 흐름 1개"란?** 앱의 가장 중요한 시나리오를 처음부터 끝까지 한 번 실행하는 것입니다. 예: "회의록 텍스트 입력 → 추출 버튼 클릭 → 액션아이템 목록 표시" — 이게 에러 없이 되면 성공입니다.

* * *

## 💡 마인드셋: 오늘 꼭 기억할 3가지

### 1. "80%면 충분하다"

> 100% 완벽한 앱은 4시간에 불가능합니다. 하지만 **80%짜리 동작하는 앱**은 가능합니다. 그리고 그 80%가 실무에서 가장 가치 있는 부분입니다.

### 2. "완벽한 기획보다 빠른 실험"

> 기획서를 3시간 쓰고 빌드를 1시간 하면 실패합니다. 기획을 30분에 끝내고 빌드를 3시간 30분 하세요. **"이런 거 되나?"를 직접 만들어서 확인**하는 게 가장 빠른 학습입니다.

### 3. "실패해도 OK"

> 프롬프트가 안 먹혀서 다시 시도하는 것도 학습입니다. 에러 메시지를 AI에게 보여주고 수정하는 루프를 반복하다 보면, 어느새 바이브코딩이 익숙해져 있을 겁니다. 오늘 중요한 건 **과정을 경험하는 것**입니다.
