# 실전 프롬프트 팁 — 두 프로젝트에서 배운 교훈

> 실제로 Replit Agent에게 프롬프트 150회 이상 주고받으며 배운, 비개발자를 위한 실전 노하우

* * *

## 🎯 이 페이지를 읽어야 하는 이유

Replit Agent는 강력하지만, **프롬프트를 어떻게 주느냐에 따라 결과가 천지차이**입니다. 이 페이지는 실제 프로젝트 2개(replit.mfitlab.com, Writer)를 만들면서 100~150회 이상의 프롬프트를 주고받은 경험에서 나온 실전 팁입니다.

이론이 아니라 **진짜 해본 사람의 교훈**입니다.

> 📎 Replit Agent의 기능(Plan Mode, Economy/Power/Turbo 모드, 체크포인트 등)에 대한 자세한 설명은 [Agent 활용팁](../2-레플릿-설명/Agent-활용팁)을 참고하세요. 이 페이지는 **프롬프트 작성법**에 집중합니다.

* * *

## 📝 실제 첫 프롬프트 원문

### mfitlab.com의 첫 프롬프트 (추정)

> "마켓핏랩의 Replit 바이브코딩 해커톤 소개 랜딩페이지를 만들어줘. React 기반으로, 해커톤 소개, 신청하기 CTA 버튼, FAQ, 프로그램 소개가 들어가는 멀티 페이지 웹사이트."

### Writer의 첫 프롬프트

> "B2B 블로그 콘텐츠 자동화 시스템을 만들어줘. 키워드를 입력하면 AI가 블로그 글을 자동으로 생성하는 웹 서비스. Python + FastAPI 기반으로, Anthropic Claude API와 Tavily 검색 API를 활용. 웹 대시보드에서 글 목록 확인, 생성, 관리가 가능해야 함."

> 💡 **포인트**: 첫 프롬프트에서 **모든 기능을 한꺼번에 요구하지 않았습니다**. 기본 구조와 핵심 기능만 먼저 요청하고, 이후에 하나씩 기능을 추가해나갔어요.

### 이후 점진적 추가 예시

```
"검수함 추가해줘. 생성된 글이 바로 발행되지 않고 승인 과정을 거치게"
"blog.replit.com 글을 크롤링해서 한국어로 번역하는 기능 추가해줘"
"SEO 점수가 80점 미만이면 자동으로 재작성하게 해줘"
"이미지를 자동으로 삽입해줘. Unsplash에서 맥락에 맞는 이미지 검색"
```

## ⭐ 프롬프트 팁 5가지 (실전 검증)

### 1. 한 번에 하나씩 요청하라

이것이 **가장 중요한 팁**입니다.

| ❌ 나쁜 프롬프트 | ✅ 좋은 프롬프트 |
|----------------|----------------|
| "글 생성이랑 번역이랑 검수함이랑 스케줄러 다 만들어줘" | "키워드를 입력하면 글을 생성하는 기본 기능을 먼저 만들어줘" |

**왜?**
- Agent가 한꺼번에 너무 많은 걸 만들면 각 기능의 완성도가 떨어짐
- 작은 기능 하나를 완성하고, 확인하고, 다음 기능을 추가하는 것이 성공률이 높음
- 문제가 생겼을 때 어디서 잘못됐는지 찾기가 쉬움

**실제 사례**: Writer 프로젝트는 19세션에 걸쳐 점진적으로 기능을 추가. 한 세션에 하나의 기능에 집중.

> ⚠️ **비용 교훈**: Writer 프로젝트에서 순수 콘텐츠 생산 비용은 $10.7 수준이었지만, 프롬프트 튜닝과 개발 시행착오에 훨씬 많은 비용이 들었습니다. 처음부터 프롬프트를 잘 구조화하면 비용을 크게 줄일 수 있다는 뜻입니다.

---

### 2. 구체적으로 지시하라 — 숫자, 형식, 조건을 명시

| ❌ 나쁜 프롬프트 | ✅ 좋은 프롬프트 |
|----------------|----------------|
| "좋은 블로그 글을 만들어줘" | "SEO 점수 80점 이상, 2,000자 이상, H2 소제목 3개 이상인 블로그 글을 생성하는 파이프라인을 만들어줘" |
| "이메일 기능 추가해줘" | "신청 폼 제출 시 담당자(cale@boxloop.ai)에게 Resend API로 이메일 알림을 보내줘. 이메일에는 신청자 이름, 이메일, 선택한 프로그램이 포함되어야 해" |
| "예쁘게 만들어줘" | "Tailwind CSS로, 모바일에서는 1열, 데스크톱에서는 3열 그리드로 카드를 배치해줘. 카드 간격은 gap-6" |

**왜?**
- "좋은", "예쁜", "깔끔한" 같은 주관적 표현은 Agent가 해석하기 어려움
- 숫자와 조건을 주면 Agent가 명확한 목표를 가지고 작업함

---

### 3. 기존 코드 맥락을 전달하라

```
✅ "이전에 만든 검수함 구조를 유지하면서, 여기에 생성일 표시를 추가해줘"
✅ "현재 /blog 페이지의 레이아웃을 유지하면서, 필터 탭을 추가해줘"
✅ "agents/writing.py의 프롬프트 구조를 참고해서, 번역용 프롬프트도 비슷한 형태로 만들어줘"
```

**왜?**
- Agent가 기존 코드 스타일과 구조를 파악하고 일관되게 수정
- 새로 처음부터 만드는 것보다 기존 코드에 추가하는 것이 안정적

---

### 4. 스크린샷을 활용하라

UI가 의도와 다르면 **스크린샷을 첨부**해서 피드백하세요:

```
✅ [스크린샷 첨부] "이 부분에서 카드가 2열로 나오는데, 3열로 변경해줘"
✅ [스크린샷 첨부] "모바일에서 이 메뉴가 잘리고 있어. 햄버거 메뉴로 바꿔줘"
✅ [스크린샷 첨부] "이 이미지가 너무 크게 표시되고 있어. 높이를 200px로 제한해줘"
```

**왜?**
- 텍스트로 UI 문제를 설명하면 오해가 생기기 쉬움
- 스크린샷은 **정확히 어디가 어떻게 잘못되었는지** 한눈에 전달
- 특히 반응형(모바일/데스크톱) 이슈에서 매우 유용

---

### 5. Plan 모드를 먼저 활용하라

큰 변경을 시작하기 전에 **Plan 모드**로 먼저 구조를 파악하게 하세요:

```
[Plan 모드]  "블로그 번역 기능을 추가하려면 어떤 파일들을 수정해야 할지 계획을 세워줘"
    ↓ 계획 확인 후
[Build 모드]  "위 계획대로 진행해줘"
```

**왜?**
- Agent가 잘못된 방향으로 작업하면 크레딧이 낭비됨
- Plan 모드는 실제 코드를 변경하지 않으므로 안전
- 계획이 마음에 안 들면 수정 지시 후 다시 Plan

> 💡 **실제 사례**: Writer 프로젝트에서 19세션 중 상당수가 Plan → Build 전환 방식으로 진행

## 😰 비개발자가 막히는 포인트 + 해결법

### 1. "API 키가 뭔지 모르겠어요"

**해결**: Replit 에디터 왼쪽 메뉴의 🔒 **Secrets** 탭에서 키 이름과 값을 붙여넣기만 하면 됩니다. 코드에서 `process.env.키이름`으로 자동으로 참조됩니다.

### 2. "환경변수? 그게 뭔데요?"

**해결**: 환경변수 = 코드에 직접 적으면 위험한 비밀번호 같은 것을 안전하게 보관하는 방법. Replit이 UI를 제공하니까 거기에 입력만 하면 됨.

### 3. "배포했는데 변경사항이 안 보여요"

**해결**: 개발 환경과 프로덕션 환경이 **분리**되어 있습니다. 코드 수정 후 반드시 **Publish** 버튼을 눌러야 프로덕션에 반영됩니다.

### 4. "에러가 나는데 뭔 말인지 모르겠어요"

**해결**: 에러 메시지를 **그대로 복사**해서 Agent에게 붙여넣으세요. "에러 났어요"보다 "TypeError: Cannot read properties of undefined (reading 'map')" 이라고 정확히 전달하면 Agent가 훨씬 빠르게 해결합니다.

### 5. "Agent가 만든 코드가 이상한데 어떻게 하죠?"

**해결**: 
- 체크포인트(자동 백업)로 이전 상태로 되돌릴 수 있음
- "방금 변경한 것 되돌려줘"라고 Agent에게 요청
- 걱정 말고 실험하세요 — 항상 되돌릴 수 있습니다

## 🔮 다시 만든다면 다르게 할 점

두 프로젝트를 만들면서 느낀, **다음에는 반드시 이렇게 하겠다**는 교훈들:

### 1. 데이터베이스를 처음부터 사용하라

- mfitlab.com은 파일시스템(마크다운, JSON)에 데이터 저장
- PostgreSQL 같은 DB를 처음부터 쓰면 검색, 필터링, 동기화가 훨씬 편함
- 재배포 시 데이터가 덮어씌워지는 문제도 해결됨

### 2. 전체 기능 목록을 먼저 정리하라

- "글 생성"만 생각하고 시작했다가 "번역", "검수", "다관점 생성" 추가하면서 구조가 복잡해짐
- 처음부터 전체 기능 목록을 Markdown으로 정리하고 시작하면 깔끔

### 3. 품질 자동 검증을 초기에 넣어라

- 번역 품질, 이미지 적합성 등을 사람이 일일이 확인하는 것은 비효율
- 자동 검증 시스템(용어 일관성 체크, 누락 감지 등)을 초기에 설계

### 4. 디자인/레이아웃 작업에 시간이 제일 많이 든다

- mfitlab.com 프로젝트에서 **디자인/레이아웃이 전체의 38%**
- 로직 구현보다 UI 미세 조정에 더 많은 크레딧이 소모됨
- **팁**: 초기에 디자인 레퍼런스(참고 사이트)를 Agent에게 전달하면 처음부터 원하는 방향에 가깝게 만들 수 있음

## ✅ 해커톤에서 바로 쓸 수 있는 체크리스트

- [ ] 첫 프롬프트에서 **핵심 기능만** 요청 (나머지는 점진적 추가)
- [ ] 요청할 때 **숫자와 조건**을 반드시 포함
- [ ] UI 문제는 **스크린샷 첨부**로 피드백
- [ ] 큰 변경 전에 **Plan 모드**로 계획 먼저
- [ ] 에러 발생 시 에러 메시지 **그대로 복사 + 붙여넣기**
- [ ] 한 번에 하나의 기능에 **집중**
- [ ] 걱정 말고 실험 — **체크포인트로 항상 되돌릴 수 있음**
