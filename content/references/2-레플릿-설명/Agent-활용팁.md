# Agent 활용팁

> Replit Agent를 200% 활용하는 핵심 기능과 전략 — 크레딧은 아끼고, 결과물은 극대화하자!

* * *

## 🎯 이 페이지를 읽어야 하는 이유

Replit Agent는 강력하지만, **아무 생각 없이 쓰면 크레딧이 순식간에 사라집니다.** 해커톤처럼 시간과 자원이 제한된 상황에서는 전략적으로 Agent를 활용하는 것이 승패를 가릅니다.

이 페이지에서는 공식 문서 기반으로 **Agent의 핵심 기능들**을 비개발자도 이해할 수 있게 정리했습니다.

> 📎 실제 프로젝트에서 배운 **프롬프트 작성 노하우**는 [실전 프롬프트 팁](../3-임팩트-만들기/실전-프롬프트-팁)을 참고하세요. 이 페이지는 **Agent 기능 활용법**에 집중합니다.

---

## ⭐ 핵심 기능 (비개발자 필수)

이 5가지는 해커톤에서 **반드시 알고 있어야 할 기능**입니다. 모르면 크레딧 낭비, 알면 효율 극대화!

### 1. Plan Mode — "먼저 계획부터 세워줘"

#### 한 줄 설명
Agent에게 바로 코드를 짜라고 하지 말고, **먼저 계획만 세우게 하는 모드**입니다.

#### 왜 필요한가?
Agent가 코드를 생성하고 수정하면 **토큰(크레딧)이 크게 소모**됩니다. Plan Mode를 켜면 Agent가 코드를 건드리지 않고 **계획만 세워주기 때문에** 토큰 소비가 훨씬 적습니다. 계획을 먼저 확인하고, 방향이 맞다고 판단되면 그때 실행하면 됩니다.

공식 문서에 따르면 Plan Mode의 핵심 장점은:

- **Chat before committing** — 작업 시작 전에 Agent와 방향을 논의할 수 있음
- **Lower token costs** — 계획 대화는 코드 생성/수정보다 토큰을 훨씬 적게 사용
- **Precision targeting** — 계획을 검토하고 조정해서 Agent가 꼭 필요한 것만 작업하게 함
- **Iterative refinement** — 토큰을 쓰기 전에 범위를 정확히 잡을 수 있음

#### 사용법
1. Agent 채팅창 하단에서 **Plan Mode** 토글을 켭니다
2. "이런 앱을 만들 건데 어떤 구조로 할지 계획 세워줘"라고 요청
3. Agent가 계획을 보여주면 검토 후 수정 요청 또는 승인
4. 계획이 확정되면 Plan Mode를 끄고 실행

#### 🏆 해커톤 팁
> **모든 새 작업은 Plan Mode로 시작하세요.** "회원가입 기능 만들어줘"라고 바로 하지 말고, "회원가입 기능을 어떤 구조로 만들지 계획 세워줘"라고 먼저 물어보세요. 방향이 틀어진 채로 코드가 짜이면 고치는 데 더 많은 크레딧이 들어갑니다.

> 💡 **공식 추천 조합**: Plan Mode + Economy Mode를 함께 사용하면 비용 절감 효과가 극대화됩니다.

### 2. Agent Modes — Economy / Power / Turbo

#### 한 줄 설명
Agent의 **성능과 크레딧 소비량**을 3단계로 조절하는 기능입니다.

#### 왜 필요한가?
모든 작업에 최고 성능이 필요한 건 아닙니다. 간단한 텍스트 수정에 Turbo 모드를 쓰는 건 스포츠카로 편의점 가는 것과 같아요. 작업 난이도에 맞춰 모드를 전환하면 크레딧을 크게 절약할 수 있습니다.

#### 각 모드 비교

| 모드 | 크레딧 소비 | 적합한 작업 | 성능 |
|------|-----------|------------|------|
| **Economy** 🟢 | 적음 | 간단한 수정, 텍스트 변경, 스타일 조정 | 기본 |
| **Power** 🟡 | 보통 | 기능 추가, 버그 수정, 일반적인 개발 작업 | 우수 |
| **Turbo** 🔴 | 많음 | 복잡한 로직, 대규모 리팩토링, 어려운 디버깅 | 최고 |

#### 사용법
1. Agent 채팅창에서 모드 선택 메뉴를 클릭
2. 현재 작업에 맞는 모드를 선택
3. 언제든지 전환 가능

#### 🏆 해커톤 팁
> **Economy → Power → Turbo 순서로 올리세요.** 처음에는 Economy로 시작하고, 결과가 만족스럽지 않을 때만 Power로 올리세요. Turbo는 정말 복잡한 작업(예: 결제 시스템 구현, 복잡한 API 연동)에만 사용하세요.

> 📊 **크레딧 절약 전략**: 해커톤 전반부(기획+기본 구조)는 Economy, 중반부(핵심 기능)는 Power, 후반부(어려운 버그 수정)에만 Turbo를 쓰면 크레딧을 효율적으로 배분할 수 있습니다.

### 3. Code Optimizations — Agent의 자기 검수 기능

#### 한 줄 설명
Agent가 코드를 작성한 뒤 **스스로 한 번 더 검토하고 실수를 잡아주는 기능**입니다.

#### 왜 필요한가?
Agent도 실수를 합니다. Code Optimizations를 켜면 Agent가 자기가 쓴 코드를 한 번 더 살펴보면서 **버그나 비효율적인 부분을 스스로 수정**합니다. 처음에 토큰이 조금 더 들더라도, **재작업(다시 시키는 것)이 줄어들기 때문에** 결과적으로는 오히려 크레딧을 절약하게 됩니다.

공식 문서에서도 다음과 같이 설명합니다:

- **ON 추천** — 대부분의 작업에서 켜두면 Agent가 자기 코드를 검토하고, 실수를 잡고, 비용이 많이 드는 재작업을 줄여줌
- **OFF 상황** — 빠른 수정이나 예산이 정말 빠듯할 때(예: 간단한 텍스트 수정)

#### 사용법
1. Agent 설정에서 **Code Optimizations** 토글 확인
2. 대부분의 경우 **ON** 상태로 두기
3. 아주 간단한 수정만 할 때는 끄기

#### 🏆 해커톤 팁
> **기본적으로 항상 ON으로 두세요.** "버튼 색깔만 바꿔줘" 같은 초간단 작업 때만 끄면 됩니다. 켜두면 Agent가 만든 코드의 품질이 올라가서 "이거 왜 안 돼?" → "다시 고쳐줘"의 반복이 줄어듭니다.

### 4. Checkpoints & Rollback — 실수 복구의 생명줄

#### 한 줄 설명
Agent가 작업할 때마다 **자동으로 저장 지점(체크포인트)을 생성**해서, 문제가 생기면 이전 상태로 되돌릴 수 있는 기능입니다.

#### 왜 필요한가?
해커톤에서 가장 무서운 상황: "아까까지 잘 되던 게 갑자기 안 돼!" Agent에게 새 기능을 시켰는데 기존 기능이 망가지는 일이 생길 수 있습니다. 이때 Checkpoint가 있으면 **"아까 그 버전으로 돌아가줘"** 한 마디로 복구할 수 있습니다.

#### 작동 방식
- Agent가 작업을 수행할 때마다 **자동으로 체크포인트가 생성**됩니다
- 각 체크포인트는 그 시점의 코드 상태를 완전히 보존합니다
- 문제가 발생하면 원하는 체크포인트로 되돌리기(Rollback) 가능
- 수동으로도 체크포인트를 만들 수 있습니다

#### 사용법
1. Agent 채팅 히스토리에서 이전 체크포인트 확인
2. 되돌리고 싶은 시점을 선택
3. "이 체크포인트로 되돌려줘"라고 요청하거나 롤백 버튼 클릭

#### 🏆 해커톤 팁
> **큰 기능을 추가하기 전에 "현재 상태 잘 기억해둬"라고 말하세요.** 만약 새 기능이 기존 것을 망가뜨리면 바로 롤백할 수 있습니다. 해커톤에서 시간은 금인데, 망가진 코드를 디버깅하는 것보다 이전 버전으로 돌아가는 게 100배 빠릅니다.

> ⚠️ **주의**: 롤백하면 해당 체크포인트 이후의 모든 변경사항이 사라집니다. 중요한 변경이 있었다면 롤백 전에 한 번 더 생각하세요.

### 5. Feedback Widget — 데모 때 실시간 피드백 받기

#### 한 줄 설명
배포된 앱에 **피드백 수집 위젯을 한 클릭으로 추가**하는 기능입니다.

#### 왜 필요한가?
해커톤 데모에서 심사위원이나 동료들의 **실시간 피드백**을 받을 수 있으면 엄청난 장점입니다. Feedback Widget을 켜면 앱 방문자가 **스크린샷과 파일을 첨부해서 피드백을 제출**할 수 있고, 이 내용이 여러분의 Inbox로 바로 전달됩니다. 심지어 **Agent에게 직접 전달**되어 바로 수정에 활용할 수도 있습니다!

#### 사용법
1. 앱을 Publishing(배포)할 때 **Enable Feedback** 토글을 켬
2. 이미 배포된 앱이라면:
   - Publishing 도구 열기
   - **Advanced Settings** 선택
   - **Enable Feedback** 토글 켜기
   - 다시 배포(Republish)
3. 앱 방문자가 피드백을 남기면 Inbox에서 확인 가능

#### 🏆 해커톤 팁
> **배포할 때 반드시 켜세요.** 데모 중에 심사위원이 "이 부분 좀 이상한데?"라고 말하면 바로 피드백 위젯으로 기록해달라고 하세요. 스크린샷까지 첨부 가능하니까 정확한 문제 파악이 가능합니다.

> 📝 **참고**: Feedback Widget은 **웹 앱에서만** 사용 가능합니다. 모바일 앱이나 자동화 프로젝트에는 적용되지 않습니다.

## 👍 유용한 기능

핵심은 아니지만 알아두면 **프로젝트 완성도를 한 단계 올려주는** 기능들입니다.

### 6. Secrets (환경변수) — API 키 안전하게 관리

#### 한 줄 설명
API 키, 비밀번호 같은 **민감한 정보를 코드에 직접 쓰지 않고 안전하게 관리**하는 기능입니다.

#### 왜 필요한가?
OpenAI API 키를 코드에 직접 적어두면 어떻게 될까요? 프로젝트를 공유하는 순간 **누구든 여러분의 API 키를 볼 수 있고, 도용할 수 있습니다.** 해커톤에서 팀원과 코드를 공유하거나, 나중에 GitHub에 올릴 때 큰 문제가 됩니다.

Secrets 기능을 쓰면 민감한 정보가 **코드와 분리**되어 안전하게 보관됩니다.

#### 사용법
1. 왼쪽 도구 모음에서 **Tools** → **Secrets** 선택
2. Key-Value 형태로 추가 (예: `OPENAI_API_KEY` = `sk-xxxxx`)
3. 코드에서는 환경변수로 접근 (예: `process.env.OPENAI_API_KEY`)
4. Agent에게 "이 API 키를 Secrets에 저장해줘"라고 말해도 됩니다

#### 🏆 해커톤 팁
> **API 키를 받자마자 바로 Secrets에 저장하세요.** 절대 코드에 직접 하드코딩하지 마세요. Agent에게 "OpenAI API 키를 써야 하는데, Secrets에서 불러오게 해줘"라고 하면 알아서 처리합니다.

### 7. Custom Domains — 전문적인 브랜딩

#### 한 줄 설명
기본 제공되는 `프로젝트명.replit.app` 주소 대신 **나만의 도메인을 연결**하는 기능입니다.

#### 왜 필요한가?
데모에서 `my-hackathon-project-12345.replit.app`보다 `www.우리팀앱.com`이 훨씬 전문적으로 보입니다. 심사위원에게도 **"진짜 서비스처럼 준비했다"**는 인상을 줄 수 있습니다.

#### 주요 특징
- **Autoscale, Reserved VM, Static** 배포 타입에서 모두 사용 가능
- **SSL/TLS 보안 인증서**를 Replit이 자동으로 제공 (무료)
- 도메인은 외부 도메인 업체에서 구매하거나 Replit에서 직접 구매 가능
- DNS 변경 후 **최대 48시간** 소요될 수 있음

#### 사용법
1. 앱을 먼저 배포(Publish)
2. Publishing 탭 → 설정(Settings)
3. **Custom Domain** 섹션에서 도메인 입력
4. 안내에 따라 DNS 레코드 설정 (CNAME 등)

#### 🏆 해커톤 팁
> 시간이 빠듯한 해커톤에서는 **DNS 반영에 최대 48시간**이 걸릴 수 있으니, 커스텀 도메인을 쓸 계획이라면 **가능한 한 빨리 설정**하세요. 아니면 기본 `.replit.app` 도메인으로도 충분히 깔끔합니다!

### 8. Scheduled Deployments — 자동으로 반복 실행

#### 한 줄 설명
정해진 시간에 **앱을 자동으로 실행**시키는 예약 배포 기능입니다. 일종의 크론잡(cron job)입니다.

#### 왜 필요한가?
"매일 아침 9시에 뉴스를 크롤링해서 정리하는 앱"이나 "매주 월요일에 리포트를 자동 생성하는 앱"같이 **주기적으로 실행되어야 하는 작업**에 딱 맞습니다.

#### 주요 특징
- **자연어로 스케줄 설정 가능** — "매일 오전 9시"라고 입력하면 AI가 자동으로 cron 표현식으로 변환
- **에러 알림** — 예약된 작업이 실패하면 알림 제공
- **모니터링** — 로그와 상태를 실시간으로 확인 가능
- 지속적인 웹 앱이 아닌 **주기적 작업(배치)** 용도에 최적화

#### 요금
- 스케줄러 자체: **무료** ($0.00)
- 실행당 기본 요금: **$1.00**/실행
- 컴퓨팅 비용: **$3.20**/compute unit (실제 실행 시간에 따라)

#### 🏆 해커톤 팁
> 해커톤 프로젝트가 **자동화/데이터 수집** 성격이라면 Scheduled Deployment를 활용하세요. "매 시간마다 특정 사이트를 체크한다"는 데모는 심사위원에게 **"실제 서비스처럼 자동화됐다"**는 강한 인상을 줍니다.

### 9. Publishing 요금 구조 — 배포 타입별 비용 이해하기

#### 한 줄 설명
Replit에서 앱을 배포할 때 **어떤 타입을 선택하느냐에 따라 비용이 달라집니다.**

#### 배포 타입 비교

| 타입 | 설명 | 적합한 프로젝트 | 비용 수준 |
|------|------|----------------|----------|
| **Static** | 정적 사이트 (HTML/CSS/JS만) | 포트폴리오, 랜딩페이지 | 💚 가장 저렴 |
| **Autoscale** | 서버 포함, 트래픽에 따라 자동 확장 | 웹앱, API 서버 | 💛 적당 |
| **Reserved VM** | 항상 켜져있는 전용 서버 | 24시간 운영 서비스 | 🔴 비쌈 |
| **Scheduled** | 정해진 시간에만 실행 | 자동화, 배치 작업 | 💚 저렴 |

#### 상세 요금 (공식 문서 기준)

**Autoscale 배포:**
- 기본 요금: **$1.00**/배포
- 요청당: **$1.20**/백만 요청
- 컴퓨팅: **$3.20**/compute unit

**Static 배포:**
- 아웃바운드 트래픽: **$0.10**/GiB

**Reserved VM 배포:**
- Small: **$40.00**/월
- Medium: **$80.00**/월
- Large: **$160.00**/월

**Scheduled 배포:**
- 실행당: **$1.00**
- 컴퓨팅: **$3.20**/compute unit

#### 어떻게 결제되나요?
모든 배포 비용은 **월간 크레딧에서 차감**됩니다. 크레딧이 다 소진된 후에만 추가 요금이 발생합니다.

- **Free Plan**: 무료 배포 1개 포함 (30일 후 만료, 재배포 가능)
- **Core Plan**: 월간 크레딧 포함
- **Pro Plan**: 티어별 월간 크레딧 포함

#### 🏆 해커톤 팁
> **대부분의 해커톤 프로젝트는 Autoscale이면 충분합니다.** 정적 사이트라면 Static으로 더 저렴하게 가능하고, Reserved VM은 해커톤에서는 거의 필요 없습니다. Replit이 프로젝트 타입에 따라 **자동으로 최적 배포 타입을 추천**해주니 그대로 따라가도 좋습니다.

> 📊 **비용 시뮬레이션**: Replit 공식 [인터랙티브 요금 계산기](https://deployment-pricing.replit.app/)에서 예상 비용을 미리 확인할 수 있습니다.

### 10. Credit Packs 가격 — 추가 크레딧 구매

#### 한 줄 설명
월간 크레딧이 부족할 때 **추가로 구매할 수 있는 크레딧 팩**입니다.

#### 가격표

| 크레딧 팩 | 가격 | 할인 |
|----------|------|------|
| $100 크레딧 | $100 | — |
| $300 크레딧 | $290 | $10 할인 |
| $500 크레딧 | $480 | $20 할인 |
| $1,000 크레딧 | $950 | $50 할인 |

#### 알아야 할 것들
- **구매 위치**: Settings → Account → Billing (Core 또는 Pro 플랜 사용자)
- **유효기간**: 구매일로부터 **6개월** 후 만료
- **자동 갱신 없음**: 자동으로 재구매되지 않음
- **이월 없음**: 만료 후 남은 크레딧은 소멸
- **자동 소비 순서**: 만료일이 가장 빠른 크레딧 팩부터 먼저 사용됨

#### 🏆 해커톤 팁
> 팀 단위 해커톤이라면 **$300 팩($290)이 가성비 최고**입니다. $10 할인이지만 6개월간 넉넉하게 쓸 수 있습니다. 해커톤 당일에 크레딧이 바닥나면 큰일이니, **여유 있게 미리 충전**해두세요.

> ⚠️ **예산 관리 팁**: Settings → Billing에서 **Usage Alert(사용량 알림)**과 **Service Shutdown Limit(서비스 중단 한도)**를 설정하면, 예상치 못한 과금을 방지할 수 있습니다.

### 11. Private Deployments — 코드 없이 접근 제한

#### 한 줄 설명
별도의 인증 코드 없이 **Replit 로그인 화면으로 앱 접근을 제한**하는 기능입니다.

#### 왜 필요한가?
아직 공개하고 싶지 않은 앱, 내부 도구, 베타 테스트 중인 앱에 누구나 접근하면 안 되겠죠? Private Deployments를 켜면 **Replit 로그인을 한 조직(Organization) 멤버만** 앱에 접근할 수 있습니다.

#### 적합한 상황
- 내부 도구 / 관리자 페이지
- 베타 테스트 중인 앱
- 민감한 데이터를 다루는 앱

#### 주의사항
- **Pro/Enterprise 구독자만** 사용 가능 *(Teams 플랜은 2026년 3월 sunset → Pro로 전환)*
- 앱이 **Team 워크스페이스** 안에 있어야 함

> 📖 **공식 문서**: [docs.replit.com/cloud-services/deployments/private-deployments](https://docs.replit.com/cloud-services/deployments/private-deployments)

### 12. App Monitoring — 배포 후 실시간 모니터링

#### 한 줄 설명
배포된 앱의 **상태, 성능, 방문자 데이터를 실시간으로 확인**하는 기능입니다.

#### 왜 필요한가?
"우리 앱에 사람들이 얼마나 접속하고 있지?" "에러가 나고 있나?" 이런 궁금증을 해결해줍니다. 특히 데모 중에 앱이 안 되는 상황을 **사전에 감지**할 수 있어요.

#### 주요 기능
- **실시간 상태 확인** — 앱이 정상 작동 중인지 한눈에 확인
- **성능 분석** — 페이지 뷰, 응답 시간, 사용자 참여 지표 모니터링
- **설정 관리** — 커맨드, Secrets, 배포 옵션을 몇 클릭으로 수정 가능

#### 사용법
1. Publishing 탭으로 이동
2. 배포된 앱의 모니터링 대시보드 확인
3. 로그, 메트릭, 상태를 실시간으로 확인

> 📖 **공식 문서**: [docs.replit.com/cloud-services/deployments/monitoring-a-deployment](https://docs.replit.com/cloud-services/deployments/monitoring-a-deployment)

## 🎯 해커톤 전략 총정리

### 크레딧 절약 3원칙

1. **Plan First** — 모든 작업은 Plan Mode로 시작
2. **Start Low** — Economy부터 시작해서 필요할 때만 올리기
3. **Optimize On** — Code Optimizations는 기본 ON

### 시간 절약 3원칙

1. **Checkpoint 활용** — 큰 변경 전에 체크포인트 확인
2. **Rollback > Debug** — 꼬였으면 되돌리기가 고치기보다 빠름
3. **Feedback Widget** — 데모 피드백을 자동화

### 추천 워크플로우

```
1. 아이디어 정리
2. Plan Mode + Economy로 구조 계획
3. Economy/Power로 핵심 기능 구현
4. Code Optimizations ON으로 품질 확보
5. Autoscale로 배포 + Feedback Widget 켜기
6. 피드백 반영하며 반복 개선
```
