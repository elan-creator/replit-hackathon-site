프롬프트: 팀 점심 메뉴를 투표로 정하는 앱 개발

1) 앱 목적
- 매일 점심시간 전에 후보 메뉴를 등록하고 팀원들이 투표하여 가장 인기 있는 메뉴를 선정하는 웹/모바일 앱을 만든다.
- 주변 맛집 정보를 네이버 지도 API로 연동하고, 이전 선택 이력을 조회할 수 있게 한다.

2) 타깃 사용자
- 회사 내 팀 단위 이용자(예시: 개발팀 10명 이상). 모든 구성원이 메뉴 후보를 등록하고 투표에 참여하는 것을 기본으로 한다.

3) 핵심 기능
- 후보 메뉴 등록: 누구나 오늘의 후보 메뉴를 등록할 수 있다.
- 투표 참여: 각 후보 메뉴에 대해 팀원이 투표 가능.
- 자동 알림 및 마감: 매일 11:30에 투표 시작 알림, 12:00에 투표 마감 및 자동 집계 → 결과를 오늘의 메뉴로 확정.
- 결과 표시: 오늘의 선정 메뉴를 한눈에 보여주는 결과 화면.
- 주변 맛집 연동: 네이버 지도 API를 이용해 오늘의 후보와 연계된 주변 맛집 정보(가까운 식당 리스트/거리, 평점 등) 표시.
- 이전 이력 조회: 과거 날짜별 선정 메뉴 및 투표 결과를 조회하는 히스토리 화면.

4) 사용자 흐름(플로우)
- 초기 로그인/회원 식별(간단한 이메일 또는 팀 계정으로 시작 가능).
- 오늘의 페이지 로드: 오늘 날짜의 후보 목록과 남은 투표 시간 표시.
- 후보 등록: 권한 없이 누구나 후보 등록 버튼으로 메뉴 입력 → 후보 목록에 즉시 반영.
- 투표: 각 후보 옆의 투표 버튼 클릭. 중복 투표 방지 로직(한 Users-Per-Day) 적용 가능.
- 마감 시나리오: 12:00에 자동으로 투표 집계 → 오늘의 메뉴 확정.
- 결과 및 이력 보기: 오늘의 메뉴와 각 후보별 득표, 과거 이력 목록 확인.
- 주변 맛집 보기: 현재 위치 또는 office 위치를 기준으로 네이버 지도 API로 주변 맛집 목록/지도 표시.
- 설정: 점심 위치(오피스 위치) 및 알림 시간 등 기본 설정 조정.

5) 데이터 구조(데이터베이스 스키마 제안)
- User(사용자)
  - id: 문자열( uuid )
  - name: 문자열
  - email: 문자열
  - avatar_url: 문자열(선택)
  - created_at: timestamp
- DayRecord(일별 기록)
  - id: 문자열
  - date: date
  - winner_menu_id: foreign key -> MenuCandidate.id
  - total_votes: 정수
- MenuCandidate(메뉴 후보)
  - id: 문자열
  - day_id: foreign key -> DayRecord.id
  - name: 문자열
  - created_by_user_id: foreign key -> User.id
  - vote_count: 정수
  - created_at: timestamp
- Vote(투표)
  - id: 문자열
  - day_id: foreign key -> DayRecord.id
  - user_id: foreign key -> User.id
  - candidate_id: foreign key -> MenuCandidate.id
  - created_at: timestamp
- Restaurant(맛집 정보) [외부 API 캐시 용]
  - id: 문자열
  - name: 문자열
  - address: 문자열
  - distance_m: 정수
  - rating: 실수
  - latitude: 실수
  - longitude: 실수
  - source: 문자열(네이버 API)
  - fetched_at: timestamp

6) API/엔드포인트 설계(예시)
- GET /api/days/{date} - 당일/특정일의 투표 상태 및 후보 목록
- POST /api/days/{date}/candidates - 후보 등록
- GET /api/days/{date}/candidates - 후보 목록 조회
- POST /api/days/{date}/votes - 투표 등록(또는 PUT/PATCH로 업데이트)
- GET /api/days/{date}/results - 그날의 결과 요약
- GET /api/restaurants/nearby - 현재 위치 기준 주변 맛집 검색
- GET /api/history - 이전 날짜의 투표 이력 목록
- POST /api/settings - 알림 시간 등 환경 설정(관리자/권한 레벨 구분 가능)

7) UI 구성(화면 구성)
- 메인 대시보드
  - 오늘의 투표 상태: 남은 시간, 남은 투표 수, 진행 상황 바
  - 후보 목록: 후보 이름, 득표 수, 투표 버튼
  - 후보 추가 버튼: 후보 등록 입력 폼 모달/인라인
  - 오늘의 결과: 확정 메뉴 표시
  - 주변 맛집 미니맵/목록: 거리, 평점, 간단 소개
- 히스토리/과거 메뉴 화면
  - 날짜별 winner, 득표수, 후보 목록 요약
- 설정 화면
  - 오피스 위치(위도/경도) 입력, 알림 시간 설정, 외부 API 키 관리(환경변수 안내)
- 반응형 UI
  - 모바일/데스크톱에서 동일하게 접근 가능하도록 레이아웃 조정
  - 간편 투표 버튼과 직관적 인포카드 디자인

8) 핵심 비즈니스 로직 및 규칙
- 누구나 오늘의 후보를 등록할 수 있다.
- 한 사용자당 하루에 한 표만 가능하도록 중복 투표 방지.
- 투표 마감 시간(12:00)에 자동으로 득표를 집계하고 winner를 확정한다.
- 후보가 하나도 없으면 자동 경고 또는 기본 안내 표시.
- 주변 맛집 정보는 네이버 지도 API를 통해 실시간으로 가져오되, API 키는 보안 환경변수로 관리.
- 과거 데이터는 히스토리로 저장되어 조회 가능.

9) 알림 및 일정
- 11:30: 투표 시작 알림(앱 내부 알림 또는 카카오톡/슬랙 웹훅 연동 옵션 제안)
- 12:00: 투표 마감 및 결과 확정 알림
- 이외 시간대에 주간/월간 요약 알림 옵션도 확장 가능

10) 보안/권한 및 개인정보
- 기본적으로 익명 후보/투표 가능하되, 필요 시 간단한 인증으로 사용자 식별
- API 키(Naver Map)의 보안 관리: 환경 변수로 저장, 프론트엔드에 노출 금지
- 데이터 백업 및 로그 레벨 설정

11) 기술 스택 제안
- 백엔드: Python (FastAPI) 또는 Node.js (Express)
- 데이터베이스: SQLite (초기 버전), 확장 시 PostgreSQL로 이관 가능
- 프런트엔드: 간단한 React/Vue 또는 서버 렌더링 HTML/CSS/JS
- 외부 연동: Naver Map API(쇼핑 포인트/UI에 지도 포함), 알림 수단(웹푸시, 이메일, KakaoTalk/SNS webhook)
- 배포/환경: Replit 환경에서 실행 가능하도록 구성, .env 파일로 API 키 관리

12) 테스트 시나리오
- 후보 등록-투표-마감 흐름 정상 동작 확인(11:30 시작, 12:00 마감)
- 중복 투표 방지 동작 확인
- 네이버 지도 API 연동 시 근처 맛집 데이터 정상 수신/표시 확인
- 이력 조회 화면에서 과거 데이터가 정확히 표시되는지 확인
- 반응형 UI의 모바일/데스크톱 레이아웃 호환성 확인

13) 확장성 고려사항
- 다수 팀의 운영 확장이 용이하도록 다중 팀/프로젝트 분리 가능 구조 설계
- 후보 등록 규칙 커스터마이즈(예: 관리자가 후보를 사전 승인하는 모드) 옵션
- 투표 시간대 커스터마이즈, 다중 언어 지원 가능성 등 향후 확장 포인트 표기

참고: Q2의 요구사항을 반영하여 초기 버전에서의 핵심 흐름(11:30 시작, 12:00 마감, 모든 사용자의 후보 등록 및 투표, 자동 집계, 네이버 맵 연동)을 충족하도록 설계합니다. API 키와 알림 연동은 보안과 운영 편의를 고려해 환경 변수/웹훅 방식으로 구현합니다.
